# 일정 참여자 선택 및 프로필 이미지 표시 기획

## 1. 개요

캘린더 MVP에 일정 참여자 기능을 추가한다. 사용자는 일정 생성/수정 시 해당 캘린더 멤버를 참여자로 선택할 수 있고, 캘린더 화면과 일정 상세에서 참여자 프로필 이미지를 확인할 수 있다.

이 기능은 Google Calendar의 참석자 경험을 참고하되, MVP에서는 초대 메일, RSVP, 외부 이메일 초대, 알림 발송 없이 "일정과 사용자 연결"에 집중한다.

## 2. 목표

- 일정 생성/수정 화면에서 참여자를 검색하고 선택할 수 있다.
- 일정 목록, 월간/주간/일간 캘린더, 일정 상세에서 참여자 프로필 이미지를 볼 수 있다.
- 참여자는 해당 캘린더에 공유된 멤버만 선택할 수 있다.
- 일정 편집 권한은 기존처럼 소속 캘린더의 `calendar_members` 기준으로 판단한다.
- 공휴일 시스템 일정에는 참여자를 추가할 수 없다.

## 3. MVP 범위

- 캘린더 멤버 검색
- 일정 생성 시 참여자 선택
- 일정 수정 시 참여자 추가/제거
- 일정 상세에서 참여자 목록 표시
- PC 월간/주간/일간 뷰에서 참여자 프로필 이미지 일부 표시
- 모바일 월간/선택 날짜 목록/상세 화면에서 참여자 프로필 이미지 표시
- 참여자 프로필 이미지가 없으면 이니셜 아바타 표시
- 참여자 수가 많으면 `+N` 형태로 축약 표시

## 4. 제외 범위

- 미가입 이메일 초대
- 외부 캘린더 참석자 연동
- 참석 여부 RSVP
- 참석자별 알림/메일/푸시 발송
- 회의실, 리소스 예약
- 참석자 권한으로 일정 편집 권한 자동 부여
- 공휴일 일정 참여자
- 프로필 이미지 업로드 UI

## 5. 핵심 사용자 시나리오

### 5.1 일정 생성 시 참여자 추가

1. 사용자가 PC 또는 모바일에서 새 일정 화면을 연다.
2. 제목, 캘린더, 시간 정보를 입력한다.
3. 참여자 입력 필드에서 이름 또는 이메일을 검색한다.
4. 검색 결과에서 사용자를 선택한다.
5. 선택된 참여자는 프로필 이미지, 이름, 이메일 chip으로 표시된다.
6. 저장하면 일정과 참여자 목록이 함께 생성된다.
7. 캘린더 일정 블록에 참여자 프로필 이미지가 표시된다.

### 5.2 일정 수정 시 참여자 변경

1. 사용자가 일정 상세에서 수정을 누른다.
2. 기존 참여자가 선택된 상태로 표시된다.
3. 사용자가 참여자를 추가하거나 제거한다.
4. 저장하면 참여자 목록이 교체된다.
5. 상세 패널과 캘린더 블록의 프로필 이미지가 갱신된다.

### 5.3 보기 권한 사용자

1. `VIEWER` 사용자가 일정을 연다.
2. 참여자 목록은 볼 수 있다.
3. 참여자 추가/삭제 UI는 비활성화된다.

## 6. UX 설계

### 6.1 PC 일정 생성/수정 모달

기존 필드 순서에서 `캘린더` 아래 또는 `장소` 위에 `참여자` 필드를 추가한다.

권장 순서:

1. 제목
2. 캘린더
3. 참여자
4. 종일 여부
5. 시작일/시간
6. 종료일/시간
7. 장소
8. 설명

참여자 필드:

- 기본 표시는 한 줄 요약 필드로 둔다.
- 요약 필드에는 24px 아바타 최대 3명과 `선택된 참여자 N명`을 표시한다.
- 우측 `선택` 버튼을 누르면 멤버 체크 리스트 패널을 연다.
- 캘린더 멤버가 8명 이상이면 패널 상단에 검색 input을 표시한다.
- 검색 input placeholder: `이름 또는 이메일로 검색`
- 검색 결과는 최대 8명 표시
- 결과 row: 프로필 이미지, 표시 이름, 이메일
- 선택된 참여자는 chip으로 표시
- chip 구성: 원형 아바타, 이름, 제거 버튼
- 자기 자신은 기본 참여자로 포함하지 않는다. 단, 상세 화면에는 생성자를 별도로 표시한다.

상태:

- 검색 결과 없음: `선택할 수 있는 캘린더 멤버가 없습니다.`
- 읽기 전용 일정: 참여자 필드 비활성화
- 공휴일 일정: `공휴일 일정에는 참여자를 추가할 수 없습니다.`

### 6.2 PC 일정 상세 패널

일정 상세 패널에 `참여자` 섹션을 추가한다.

표시:

- 제목: `참여자`
- 1~5명: 아바타 + 이름 + 이메일 목록
- 6명 이상: 처음 5명 표시 후 `외 N명`
- 프로필 이미지는 32px 원형
- 이미지가 없으면 이름 첫 글자 기반 이니셜 아바타 표시

### 6.3 PC 월간 뷰

월간 셀의 일정 라벨 오른쪽 끝에 참여자 아바타를 작게 표시한다.

정책:

- 셀 공간이 좁으므로 최대 2명만 표시
- 3명 이상이면 `+N`을 16~18px 원형 badge로 표시
- 일정 제목이 잘리지 않도록 아바타 영역은 고정 폭으로 둔다.
- 공휴일 일정에는 참여자 아바타를 표시하지 않는다.

### 6.4 PC 주간/일간 뷰

시간 블록 내부 하단 또는 우측 상단에 참여자 아바타 stack을 표시한다.

정책:

- 블록 높이가 48px 미만이면 아바타 숨김
- 블록 높이가 충분하면 최대 3명 표시
- hover 시 tooltip: `김민수, 이지혜 외 2명`

### 6.5 모바일 일정 생성/수정

모바일에서는 참여자 검색을 별도 섹션으로 제공한다.

구성:

- label: `참여자`
- 참여자 행을 탭하면 `참여자 선택` 전체화면 시트를 연다.
- 시트 상단에는 제목과 `완료` 버튼을 둔다.
- 본문은 체크박스 목록으로 구성한다.
- 캘린더 멤버가 8명 이상이면 상단에 검색 input을 표시한다.
- 하단 고정 영역에 `선택한 N명`과 아바타 요약을 표시한다.
- chip 제거 터치 영역은 최소 36px

모바일에서는 화면 밀도를 위해 선택된 참여자 chip에는 `아바타 + 이름`만 표시하고, 이메일은 검색 결과에서만 표시한다.

### 6.6 모바일 캘린더/일정 목록

- 월간 달력 셀에는 기존처럼 일정 점 표시 중심을 유지한다.
- 선택 날짜 일정 목록 카드에는 참여자 아바타 최대 3명 표시
- 일정 상세 화면에는 전체 참여자 목록 표시
- 모바일 월간 셀에는 아바타를 직접 넣지 않는다. 셀이 작아 날짜/점과 겹칠 위험이 크다.

## 7. 데이터 모델

### 7.1 app_users 확장

기존 `app_users`에 프로필 이미지 URL을 추가한다.

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| profile_image_url | varchar(1000) | N | 프로필 이미지 URL |
| avatar_color | varchar(7) | N | 이미지가 없을 때 이니셜 아바타 배경색 |

MVP에서는 이미지 업로드를 구현하지 않고 seed 데이터 또는 관리자 입력 URL을 사용한다. 이미지가 없으면 프론트에서 `display_name` 첫 글자로 이니셜 아바타를 생성한다.

### 7.2 event_participants

일정과 참여자를 연결하는 테이블이다.

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| id | uuid | Y | 참여자 ID |
| event_id | uuid | Y | 일정 ID |
| user_id | uuid | Y | 참여자 사용자 ID |
| added_by | uuid | N | 참여자를 추가한 사용자 |
| response_status | varchar(30) | Y | `NEEDS_ACTION`, `ACCEPTED`, `DECLINED`, `TENTATIVE` |
| created_at | timestamptz | Y | 생성 시각 |
| updated_at | timestamptz | Y | 수정 시각 |
| deleted_at | timestamptz | N | 소프트 삭제 시각 |

MVP 정책:

- `response_status`는 우선 `NEEDS_ACTION`으로 저장한다.
- RSVP UI는 만들지 않는다.
- 같은 일정에 같은 사용자는 1번만 추가할 수 있다.
- 삭제는 soft delete를 권장한다.

제약:

- unique partial `(event_id, user_id) where deleted_at is null`
- `event_id`는 `events.id` 참조
- `user_id`는 `app_users.id` 참조

인덱스:

- `(event_id)`
- `(user_id)`
- `(user_id, created_at desc)`

## 8. API 설계

Base URL은 기존과 동일하게 `/api/v1`을 사용한다.

### 8.1 캘린더 멤버 검색 API

`GET /api/v1/calendars/{calendarId}/members/search?query=김&limit=8`

목적:

- 일정 참여자 선택용 캘린더 멤버 검색

권한:

- 로그인 필요
- 대상 캘린더 `VIEWER` 이상

쿼리:

- `query`: 이름 또는 이메일 일부, 2자 이상
- `limit`: 기본 8, 최대 20

응답:

```json
{
  "data": [
    {
      "id": "user-id",
      "email": "viewer@example.com",
      "displayName": "이지혜",
      "profileImageUrl": "https://example.com/avatar.jpg",
      "avatarColor": "#8B5CF6"
    }
  ],
  "meta": {}
}
```

제외:

- 삭제된 사용자
- 대상 캘린더 멤버가 아닌 사용자
- 자기 자신은 검색 결과에 표시할 수 있지만, 기본 선택은 하지 않는다.

### 8.2 일정 생성 API 확장

`POST /api/v1/events`

요청에 `participantUserIds`를 추가한다.

```json
{
  "calendarId": "calendar-id",
  "title": "회의",
  "description": "주간 회의",
  "location": "회의실 A",
  "startsAt": "2026-05-24T10:00:00+09:00",
  "endsAt": "2026-05-24T11:00:00+09:00",
  "timezone": "Asia/Seoul",
  "isAllDay": false,
  "participantUserIds": ["user-id-1", "user-id-2"]
}
```

처리:

1. 일정 생성 권한 확인
2. 캘린더 읽기 전용 여부 확인
3. 참여자 사용자가 해당 캘린더 멤버인지 확인
4. 중복 참여자 제거
5. 일정 생성
6. `event_participants` 생성

### 8.3 일정 수정 API 확장

`PATCH /api/v1/events/{eventId}`

요청에 `participantUserIds`를 추가한다.

정책:

- 요청에 `participantUserIds`가 포함되면 기존 참여자 목록을 요청 값으로 교체한다.
- 요청에 `participantUserIds`가 없으면 참여자 목록은 변경하지 않는다.
- 공휴일 일정은 참여자 수정 불가
- 읽기 전용 캘린더 일정은 참여자 수정 불가

### 8.4 일정 조회 응답 확장

목록 API와 상세 API의 응답 크기를 분리한다.

`GET /api/v1/events` 목록 응답에는 `participantPreview`, `participantCount`를 추가한다.

```json
{
  "id": "event-id",
  "calendarId": "calendar-id",
  "title": "회의",
  "startsAt": "2026-05-24T10:00:00+09:00",
  "endsAt": "2026-05-24T11:00:00+09:00",
  "isAllDay": false,
  "calendarColor": "#2563EB",
  "calendarName": "업무",
  "participantCount": 4,
  "participantPreview": [
    {
      "userId": "user-id-1",
      "email": "viewer@example.com",
      "displayName": "이지혜",
      "profileImageUrl": null,
      "avatarColor": "#8B5CF6"
    }
  ]
}
```

`GET /api/v1/events/{eventId}` 상세 응답에는 전체 `participants`를 추가한다.

```json
{
  "id": "event-id",
  "calendarId": "calendar-id",
  "title": "회의",
  "startsAt": "2026-05-24T10:00:00+09:00",
  "endsAt": "2026-05-24T11:00:00+09:00",
  "isAllDay": false,
  "calendarColor": "#2563EB",
  "calendarName": "업무",
  "participantCount": 1,
  "participants": [
    {
      "userId": "user-id-1",
      "email": "viewer@example.com",
      "displayName": "이지혜",
      "profileImageUrl": null,
      "avatarColor": "#8B5CF6",
      "responseStatus": "NEEDS_ACTION"
    }
  ]
}
```

목록 조회 최적화:

- 월간/주간/일간 목록 응답은 일정당 최대 5명의 `participantPreview`만 포함한다.
- 상세 API는 전체 참여자 목록을 내려준다.

권장 응답 필드:

- `participantPreview`: 목록 미리보기용 최대 5명
- `participants`: 상세 전체 목록
- `participantCount`: 전체 참여자 수

### 8.5 캘린더 변경 시 참여자 처리

MVP에서는 일정의 캘린더 이동을 제외하므로 일반 수정 화면에서 캘린더를 바꿀 수 없다. 추후 캘린더 이동 기능을 추가할 경우, 기존 참여자 중 새 캘린더 멤버가 아닌 사용자는 저장 전 확인을 거쳐 제외한다.

권장 안내:

- `선택한 참여자 중 2명은 새 캘린더 멤버가 아니어서 제외됩니다.`

## 9. 권한 정책

참여자 추가/삭제 권한은 일정 편집 권한과 동일하게 판단한다.

| 권한 | 참여자 조회 | 참여자 추가/삭제 | 비고 |
| --- | --- | --- | --- |
| VIEWER | 가능 | 불가 | 읽기 전용 |
| EDITOR | 가능 | 가능 | 일정 편집 가능 |
| ADMIN | 가능 | 가능 | 일정 편집 가능 |
| OWNER | 가능 | 가능 | 전체 권한 |

정책:

- 참여자는 대상 캘린더의 기존 멤버만 선택할 수 있다.
- 참여자로 추가되었다고 해서 캘린더 권한이 변경되지는 않는다.
- 캘린더 공유와 일정 참여자는 별개 개념이다.
- 캘린더 멤버에서 제거된 사용자는 이후 일정 참여자로 다시 선택할 수 없다.
- 멤버 제거 시 기존 참여자 기록은 soft delete 처리하는 것을 권장한다.

MVP에서 혼란을 줄이기 위한 UI 문구:

- `참여자는 이 캘린더에 공유된 사용자 중에서 선택할 수 있습니다.`

## 10. 입력 검증

- `participantUserIds`는 배열이어야 한다.
- 최대 참여자 수는 일정당 20명으로 제한한다.
- 존재하지 않는 사용자 ID는 `VALIDATION_ERROR`
- 삭제된 사용자 ID는 `VALIDATION_ERROR`
- 대상 캘린더 멤버가 아닌 사용자 ID는 `FORBIDDEN`
- 중복 ID는 서버에서 제거한다.
- 공휴일 일정에 참여자 요청이 있으면 `READONLY_CALENDAR`
- 보기 권한 사용자의 참여자 변경은 `FORBIDDEN`
- 사용자 검색 query는 trim 후 2자 이상이어야 한다.

## 11. 에러 처리

추가 에러 코드:

- `PARTICIPANT_NOT_FOUND`: 참여자 사용자를 찾을 수 없음
- `PARTICIPANT_NOT_CALENDAR_MEMBER`: 참여자가 대상 캘린더 멤버가 아님
- `PARTICIPANT_LIMIT_EXCEEDED`: 참여자 수 제한 초과

기존 공통 실패 응답 포맷을 그대로 사용한다.

```json
{
  "error": {
    "code": "PARTICIPANT_LIMIT_EXCEEDED",
    "message": "참여자는 최대 20명까지 추가할 수 있습니다.",
    "details": []
  }
}
```

## 12. 프로필 이미지 표시 정책

이미지 우선순위:

1. `profileImageUrl`
2. `displayName` 첫 글자 이니셜
3. 이메일 첫 글자 이니셜

스타일:

- 원형 아바타
- PC 월간 라벨: 18px
- PC 상세/검색 결과: 32px
- 모바일 카드: 28px
- 겹침 stack은 `-space-x-2` 수준으로 최소화
- 이미지 로딩 실패 시 이니셜 아바타로 fallback

접근성:

- 이미지 alt: `{displayName} 프로필`
- 이니셜 아바타 aria-label: `{displayName} 프로필`
- 이벤트 버튼 aria-label에는 `참여자 N명`을 포함한다.
- 참여자 선택 목록은 `Tab` 이동과 `Space` 선택을 지원한다.

## 13. 수용 기준

- 일정 생성 시 캘린더 멤버를 검색해 참여자로 선택할 수 있다.
- 일정 저장 후 상세 화면에서 참여자 목록이 보인다.
- 일정 수정 시 기존 참여자가 선택된 상태로 보이고, 추가/제거 후 저장할 수 있다.
- 월간/주간/일간 PC 캘린더에서 일정별 참여자 아바타가 표시된다.
- 모바일 선택 날짜 일정 목록과 상세 화면에서 참여자 아바타가 표시된다.
- 참여자 3명 이상은 `+N`으로 축약 표시된다.
- 프로필 이미지가 없는 사용자는 이니셜 아바타로 표시된다.
- VIEWER는 참여자를 볼 수 있지만 수정할 수 없다.
- 공휴일 일정에는 참여자를 추가할 수 없다.
- 참여자 추가가 캘린더 공유 권한을 자동 변경하지 않는다.

## 14. 주요 리스크

- 캘린더 멤버와 일정 참여자의 차이를 사용자가 헷갈릴 수 있다.
- 월간 셀에 아바타가 과하게 들어가면 일정 제목 가독성이 떨어질 수 있다.
- 목록 API에서 참여자 전체를 매번 조인하면 월간 조회 성능이 떨어질 수 있다.
- 프로필 이미지 외부 URL이 깨질 수 있다.

대응:

- 참여자는 대상 캘린더 멤버 중에서만 선택하게 하고, 저장 화면에 안내 문구를 표시한다.
- 월간 뷰는 최대 2명만 표시하고 나머지는 `+N` 처리한다.
- 목록 API는 미리보기 참여자 수를 제한하고 상세 API에서 전체를 조회한다.
- 이미지 로딩 실패 fallback을 필수 구현한다.

## 16. 오픈 결정 사항

1. 자기 자신을 참여자에 포함할지, 생성자로만 표시할지 결정이 필요하다.
2. 프로필 이미지 업로드를 언제 도입할지 결정이 필요하다.
3. RSVP를 다음 단계에서 구현할지, 알림 기능 이후로 미룰지 결정이 필요하다.
4. 일정당 참여자 수 제한 20명이 적절한지 운영 기준 확정이 필요하다.
5. 캘린더 멤버 제거 시 기존 일정 참여자 기록을 즉시 soft delete할지, 상세 화면에서 비활성 사용자로 표시할지 결정이 필요하다.
