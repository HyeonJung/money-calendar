# 캘린더 서비스 MVP 기획서

작성일: 2026-05-24  
버전: 0.1  
상태: MVP 기획 초안

## 1. 프로젝트 개요

사용자가 개인 일정, 업무 일정, 가족 일정처럼 목적이 다른 여러 캘린더를 만들고 관리할 수 있는 캘린더 서비스를 만든다. 사용자는 각 캘린더의 이름, 색상, 설명, 공개상태를 설정할 수 있으며, 필요한 캘린더를 다른 사용자와 공유할 수 있다.

MVP는 Google Calendar와 유사한 사용 경험을 참고하되, 빠른 1차 구현을 위해 범위를 단순화한다. 핵심은 여러 캘린더 관리, 캘린더별 일정 CRUD, 캘린더 공유 권한, PC/모바일 캘린더 화면, 대한민국 공휴일 자동 세팅이다.

대한민국 공휴일은 사용자가 직접 만들지 않아도 기본으로 보이는 읽기 전용 시스템 캘린더로 제공한다. 공휴일 데이터는 배치로 수집하고, 배치 실행 이력과 실패 로그를 저장해 운영자가 상태를 추적할 수 있게 한다.

MVP의 목표는 다음과 같다.

1. 사용자가 여러 캘린더를 목적별로 생성하고 구분해 볼 수 있다.
2. 캘린더 단위로 공유 권한을 부여할 수 있다.
3. 캘린더별 일정 생성, 조회, 수정, 삭제가 가능하다.
4. PC와 모바일에서 월/주/일 중심의 기본 일정 관리를 할 수 있다.
5. 대한민국 공휴일이 기본 캘린더로 자동 제공된다.
6. 공휴일 배치 운영 상태를 데이터로 확인할 수 있다.

## 2. MVP 범위

### 2.1 포함 범위

#### 캘린더 관리

- 사용자는 여러 캘린더를 생성할 수 있다.
- 사용자는 본인이 소유하거나 관리 권한을 가진 캘린더를 수정할 수 있다.
- 캘린더 필드는 다음을 포함한다.
  - 이름
  - 색상
  - 설명
  - 공개상태
- 사용자는 캘린더를 삭제할 수 있다.
- 사용자는 캘린더 표시 여부를 켜고 끌 수 있다.
- 기본 개인 캘린더는 회원 가입 또는 최초 접속 시 자동 생성한다.
- 공휴일 캘린더는 모든 사용자에게 기본 구독 상태로 제공한다.

#### 캘린더 공유

- 사용자는 본인 캘린더를 다른 사용자와 공유할 수 있다.
- 공유 권한은 보기, 편집, 관리자 3단계로 제공한다.
- 캘린더 소유자는 공유 사용자를 추가, 권한 변경, 제거할 수 있다.
- 관리자 권한 사용자는 소유자와 유사하게 공유 설정을 관리할 수 있으나, 소유자를 제거할 수 없다.
- 공유받은 캘린더는 내 캘린더 목록에 표시된다.

#### 일정 관리

- 사용자는 권한이 있는 캘린더에 일정을 생성할 수 있다.
- 일정 생성 시 캘린더를 선택한다.
- 일정 필드는 다음을 포함한다.
  - 제목
  - 시작일시
  - 종료일시
  - 종일 여부
  - 설명
  - 장소
  - 캘린더
- 사용자는 권한에 따라 일정을 조회, 수정, 삭제할 수 있다.
- 일정은 월간, 주간, 일간 화면에서 조회할 수 있다.
- 일정 색상은 기본적으로 소속 캘린더 색상을 따른다.

#### PC 화면

- PC용 캘린더 페이지를 제공한다.
- 좌측에는 미니 캘린더와 캘린더 목록을 둔다.
- 중앙에는 월/주/일 캘린더 뷰를 둔다.
- 우측 또는 모달에는 일정 상세/편집 패널을 둔다.
- 일정 생성, 수정, 삭제, 캘린더 표시 토글, 뷰 전환을 지원한다.

#### 모바일 화면

- 모바일용 캘린더 페이지를 제공한다.
- 작은 화면에 맞춰 상단 날짜 네비게이션, 하단 액션, 일정 목록 중심의 UX를 제공한다.
- 월간 화면에서는 일정 밀도를 줄이고 선택한 날짜의 일정 목록을 하단에 표시한다.
- 일정 생성/수정은 전체 화면 바텀시트 또는 별도 화면으로 제공한다.

#### 대한민국 공휴일

- 대한민국 공휴일은 배치로 수집한다.
- 수집된 공휴일은 읽기 전용 시스템 캘린더에 자동 반영한다.
- 모든 사용자는 공휴일 캘린더를 기본 구독한다.
- 사용자는 공휴일 캘린더의 표시 여부만 변경할 수 있다.
- 사용자는 공휴일 캘린더 자체를 삭제하거나 수정할 수 없다.
- 배치 실행 이력과 실패 로그를 저장한다.

### 2.2 MVP 성공 기준

- 신규 사용자가 기본 개인 캘린더와 대한민국 공휴일 캘린더를 즉시 볼 수 있다.
- 사용자가 개인 캘린더를 2개 이상 만들고 색상으로 구분할 수 있다.
- 사용자가 특정 캘린더를 다른 사용자에게 보기/편집/관리자 권한으로 공유할 수 있다.
- 편집 권한 이상 사용자가 공유 캘린더에 일정을 만들고 수정할 수 있다.
- 보기 권한 사용자는 공유 캘린더의 일정을 수정하거나 삭제할 수 없다.
- 공휴일 배치가 실행 결과와 실패 사유를 남긴다.

## 3. 제외 범위

MVP에서 제외할 기능은 다음과 같다.

- 반복 일정
- 참석자 초대와 RSVP
- 알림, 푸시, 이메일 리마인더
- Google Calendar, Apple Calendar, Outlook 외부 연동
- iCal 구독 URL 발급
- 캘린더 가져오기/내보내기
- 일정 첨부파일
- 회의실, 리소스 예약
- 위치 기반 지도 연동
- 일정 검색 고도화
- 자연어 일정 입력
- 타임존 자동 변환 고도화
- 공개 캘린더 디렉터리
- 조직/팀 단위 관리자 콘솔
- 감사 로그 전체 UI
- 일정 변경 이력 복원
- 드래그 앤 드롭 일정 이동
- 반복 공휴일 규칙 직접 계산
- 음력 일정 직접 생성

단, 데이터 모델은 향후 반복 일정, 참석자, 알림을 추가할 수 있도록 지나치게 막히지 않는 형태로 설계한다.

## 4. 핵심 사용자

### 4.1 개인 사용자

개인 일정, 업무 일정, 가족 일정처럼 여러 목적의 일정을 구분해서 관리하려는 사용자다. 캘린더별 색상과 표시 토글을 이용해 필요한 일정만 보고 싶어 한다.

주요 니즈:

- 캘린더를 목적별로 나누기
- 빠른 일정 생성
- 월/주/일 화면에서 일정 확인
- 공휴일 자동 확인

### 4.2 공유 캘린더 사용자

가족, 동료, 프로젝트 멤버와 특정 캘린더를 함께 쓰는 사용자다. 누군가는 보기만 하고, 누군가는 일정을 추가하거나 수정해야 한다.

주요 니즈:

- 다른 사용자에게 캘린더 공유
- 사용자별 권한 제어
- 공유받은 캘린더의 일정 확인
- 권한에 맞는 안전한 수정

### 4.3 운영자

대한민국 공휴일 배치 수집 상태를 확인하고 실패 원인을 파악해야 하는 사용자다. MVP에서는 별도 운영자 UI가 없더라도 데이터와 로그가 남아야 한다.

주요 니즈:

- 공휴일 배치 실행 결과 확인
- 실패 사유와 원본 응답 추적
- 특정 연도 재수집
- 중복 공휴일 방지

## 5. 사용자 시나리오

### 5.1 첫 접속

1. 사용자가 로그인 후 캘린더 페이지에 처음 진입한다.
2. 시스템은 사용자 기본 개인 캘린더가 있는지 확인한다.
3. 기본 개인 캘린더가 없으면 `내 캘린더`를 생성한다.
4. 시스템은 대한민국 공휴일 시스템 캘린더 구독 상태가 있는지 확인한다.
5. 구독 상태가 없으면 공휴일 캘린더를 표시 상태로 추가한다.
6. 사용자는 현재 월 화면에서 내 일정과 공휴일을 함께 본다.

### 5.2 새 캘린더 생성

1. 사용자가 좌측 캘린더 목록에서 캘린더 추가를 선택한다.
2. 이름, 색상, 설명, 공개상태를 입력한다.
3. 저장을 누른다.
4. 시스템은 이름 중복 여부와 색상 형식을 검증한다.
5. 캘린더가 생성되고 목록에 추가된다.
6. 사용자는 해당 캘린더 표시 여부를 토글할 수 있다.

### 5.3 일정 생성

1. 사용자가 캘린더 화면에서 날짜 또는 시간 슬롯을 선택한다.
2. 일정 생성 모달 또는 바텀시트가 열린다.
3. 사용자는 제목, 시작일시, 종료일시, 종일 여부, 캘린더, 설명, 장소를 입력한다.
4. 시스템은 사용자가 선택한 캘린더에 편집 권한 이상을 갖는지 확인한다.
5. 시스템은 시작일시와 종료일시가 유효한지 검증한다.
6. 일정이 저장된다.
7. 캘린더 화면에 새 일정이 캘린더 색상으로 표시된다.

### 5.4 일정 수정

1. 사용자가 기존 일정을 선택한다.
2. 일정 상세 패널이 열린다.
3. 사용자가 편집 권한 이상이면 수정 버튼을 볼 수 있다.
4. 사용자가 제목, 시간, 설명, 장소 등을 수정한다.
5. 시스템은 권한과 입력값을 검증한다.
6. 수정된 일정이 저장된다.
7. 목록과 캘린더 뷰가 갱신된다.

### 5.5 일정 삭제

1. 사용자가 기존 일정을 선택한다.
2. 일정 상세 패널에서 삭제를 선택한다.
3. 시스템은 편집 권한 이상인지 확인한다.
4. 삭제 확인 메시지를 표시한다.
5. 사용자가 확인하면 일정을 삭제한다.
6. 캘린더 화면에서 일정이 사라진다.

### 5.6 캘린더 공유

1. 캘린더 소유자가 캘린더 설정에서 공유 탭을 연다.
2. 공유할 사용자의 이메일 또는 사용자 식별자를 입력한다.
3. 권한을 보기, 편집, 관리자 중 하나로 선택한다.
4. 시스템은 대상 사용자가 존재하는지 확인한다.
5. 시스템은 이미 공유된 사용자라면 기존 권한 변경으로 처리한다.
6. 공유가 완료되면 대상 사용자의 캘린더 목록에 표시된다.
7. 대상 사용자는 권한에 맞게 일정을 조회하거나 수정한다.

### 5.7 공휴일 배치 수집

1. 스케줄러가 정해진 시간에 대한민국 공휴일 수집 배치를 실행한다.
2. 시스템은 대상 연도를 계산한다.
3. 외부 공휴일 데이터 소스에서 대상 연도의 공휴일 목록을 조회한다.
4. 원본 데이터를 내부 공휴일 형식으로 정규화한다.
5. 같은 날짜와 이름의 공휴일이 이미 있으면 갱신하고, 없으면 추가한다.
6. 공휴일 시스템 캘린더의 이벤트로 반영한다.
7. 실행 결과, 수집 건수, 생성 건수, 갱신 건수, 실패 건수, 실행 시간을 저장한다.
8. 실패하면 실패 사유, 단계, 외부 응답 요약을 저장한다.

## 7. PC 화면 설계

### 7.1 PC 레이아웃 원칙

- 캘린더 본문을 가장 넓게 배치한다.
- 캘린더 목록은 좌측 사이드바에 고정한다.
- 일정 생성과 상세 확인은 모달 또는 우측 패널로 처리한다.
- 월/주/일 전환은 상단에서 항상 접근 가능해야 한다.
- PC에서는 정보 밀도를 높이되, 색상과 여백으로 일정 구분을 명확히 한다.

### 7.2 PC 캘린더 페이지

#### 상단 바

요소:

- 오늘 버튼
- 이전/다음 버튼
- 현재 기간 텍스트
  - 월간: `2026년 5월`
  - 주간: `2026.05.18 - 2026.05.24`
  - 일간: `2026년 5월 24일`
- 뷰 전환 segmented control
  - 월
  - 주
  - 일
- 새 일정 버튼

동작:

- 오늘 버튼 클릭 시 오늘 날짜가 포함된 화면으로 이동한다.
- 이전/다음 버튼은 현재 뷰 단위로 이동한다.
- 새 일정 버튼은 기본 캘린더가 선택된 일정 생성 모달을 연다.

#### 좌측 사이드바

영역:

- 미니 월간 달력
- 내 캘린더
- 공유받은 캘린더
- 시스템 캘린더
- 캘린더 추가 버튼

캘린더 항목 표시:

- 색상 점 또는 체크박스
- 캘린더 이름
- 권한 배지
  - 소유자
  - 보기
  - 편집
  - 관리자
- 더보기 메뉴

항목 동작:

- 체크박스 클릭 시 화면 표시 여부를 변경한다.
- 이름 클릭 시 해당 캘린더 설정을 연다.
- 더보기 메뉴에서 설정, 공유, 삭제를 제공한다.
- 공휴일 캘린더는 설정 진입 시 읽기 전용 안내를 보여준다.

#### 월간 뷰

구성:

- 7열 그리드
- 일요일부터 토요일까지 표시
- 이전/다음 달의 날짜는 흐리게 표시
- 오늘은 강조 표시
- 선택한 날짜는 테두리 또는 배경으로 표시
- 각 날짜 셀에 일정 목록 표시

일정 표시:

- 종일 일정은 날짜 셀 상단에 표시한다.
- 시간 일정은 시작 시간과 제목을 표시한다.
- 일정 색상은 캘린더 색상을 사용한다.
- 한 날짜에 일정이 많으면 최대 3개까지 표시하고 `+N개`로 접는다.

동작:

- 빈 날짜 셀 클릭 시 해당 날짜로 종일 일정 생성 모달을 연다.
- 일정 클릭 시 상세 패널을 연다.
- `+N개` 클릭 시 해당 날짜의 일정 목록 팝오버를 연다.

#### 주간 뷰

구성:

- 좌측 시간축
- 7일 열
- 상단 종일 일정 영역
- 시간 단위 그리드

시간 단위:

- MVP 기본 단위는 30분이다.
- 화면 표시는 1시간 라벨을 기준으로 하고, 내부 선택은 30분 단위를 지원한다.

동작:

- 시간 슬롯 클릭 시 해당 시작 시간으로 일정 생성 모달을 연다.
- 일정 클릭 시 상세 패널을 연다.
- 종일 일정은 상단 영역에 표시한다.

#### 일간 뷰

구성:

- 단일 날짜
- 상단 종일 일정 영역
- 시간표
- 우측 또는 하단 일정 요약 목록

동작:

- 시간 슬롯 클릭으로 일정 생성
- 일정 클릭으로 상세 확인
- 일정이 없는 경우 빈 상태 문구를 표시한다.

#### 일정 생성/수정 모달

필드:

- 제목
- 캘린더
- 종일 여부
- 시작일
- 시작시간
- 종료일
- 종료시간
- 장소
- 설명

버튼:

- 저장
- 취소
- 삭제

권한별 상태:

- 보기 권한: 입력 필드 비활성화, 삭제 버튼 숨김
- 편집 권한: 일정 생성/수정/삭제 가능
- 관리자 권한: 일정 생성/수정/삭제 가능
- 공휴일 캘린더: 항상 읽기 전용

### 7.3 PC 캘린더 설정 화면

#### 기본 정보 탭

필드:

- 이름
- 색상
- 설명
- 공개상태

공개상태 옵션:

- 비공개
- 링크 접근 가능
- 공개

MVP 기본값은 `비공개`다. `링크 접근 가능`과 `공개`는 데이터 필드와 설정 UI만 준비하고, 공개 목록 노출이나 검색 노출은 제외할 수 있다.

#### 공유 탭

구성:

- 현재 공유 사용자 목록
- 사용자 이메일
- 권한 선택
- 공유 추가 버튼

목록 컬럼:

- 사용자 이름
- 이메일
- 권한
- 공유된 시각
- 작업

작업:

- 권한 변경
- 공유 제거

제약:

- 소유자는 제거할 수 없다.
- 보기 권한 사용자는 공유 탭을 볼 수 없다.
- 편집 권한 사용자는 공유 탭을 볼 수 있으나 변경할 수 없다.
- 관리자 권한 사용자는 소유자 변경을 제외하고 공유 설정을 변경할 수 있다.

## 8. Mobile 화면 설계

### 8.1 모바일 레이아웃 원칙

- 월간 화면은 날짜 탐색에 집중하고, 일정 상세는 선택 날짜 목록으로 제공한다.
- 작은 화면에서는 좌측 사이드바를 쓰지 않고 캘린더 목록을 필터 바텀시트로 제공한다.
- 주요 액션은 하단 고정 버튼 또는 플로팅 버튼으로 제공한다.
- 일정 생성/수정은 한 화면에 한 맥락만 보여준다.

### 8.2 모바일 캘린더 페이지

#### 상단 영역

요소:

- 이전/다음 버튼
- 현재 기간 텍스트
- 오늘 버튼
- 뷰 전환 버튼
- 캘린더 필터 버튼

뷰 전환:

- 월
- 주
- 일

모바일 기본 진입 뷰는 `월`이다.

#### 월간 뷰

구성:

- 7열 축약 월간 달력
- 날짜 숫자
- 일정 존재 여부 점 표시
- 오늘 강조
- 선택 날짜 강조
- 선택 날짜 일정 목록

일정 표시:

- 날짜 셀 안에는 일정 제목을 많이 넣지 않는다.
- 캘린더 색상 점을 최대 3개까지 표시한다.
- 선택한 날짜 하단에 일정 목록을 카드 또는 리스트로 표시한다.

동작:

- 날짜 탭 시 선택 날짜가 바뀐다.
- 일정 목록 항목 탭 시 상세 화면을 연다.
- 플로팅 추가 버튼 탭 시 선택 날짜 기준으로 일정 생성 화면을 연다.

#### 주간 뷰

구성:

- 상단 7일 스트립
- 선택 날짜의 일정 목록
- 필요 시 시간순 리스트

MVP 모바일 주간 뷰는 PC처럼 복잡한 시간표를 그대로 축소하지 않는다. 7일 스트립에서 날짜를 고르고, 아래에는 해당 날짜 일정을 시간순으로 보여준다.

#### 일간 뷰

구성:

- 날짜 헤더
- 종일 일정 섹션
- 시간 일정 리스트
- 빈 상태

동작:

- 시간 일정은 시작 시간 기준으로 정렬한다.
- 일정 항목에는 제목, 시간, 캘린더 색상, 장소를 표시한다.

### 8.3 모바일 캘린더 필터

캘린더 필터는 바텀시트로 제공한다.

구성:

- 내 캘린더
- 공유받은 캘린더
- 시스템 캘린더
- 전체 선택
- 전체 해제

각 항목:

- 색상 점
- 이름
- 권한 배지
- 표시 토글

### 8.4 모바일 일정 생성/수정

일정 생성/수정은 전체 화면 또는 바텀시트로 제공한다.

필드 순서:

1. 제목
2. 캘린더
3. 종일 여부
4. 시작일/시간
5. 종료일/시간
6. 장소
7. 설명

하단 버튼:

- 저장
- 취소

### 8.5 모바일 캘린더 설정

설정은 별도 화면으로 제공한다.

탭 또는 섹션:

- 공유
- 표시 설정

## 9. 데이터 모델

아래 모델은 관계형 데이터베이스 기준의 권장 설계다. 실제 DB에 맞춰 타입과 제약 조건은 조정할 수 있다.

### 9.1 users

사용자 계정 테이블이다. 인증 시스템이 이미 있다면 기존 사용자 테이블을 사용한다.

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| id | uuid | Y | 사용자 ID |
| email | varchar(255) | Y | 로그인 이메일 |
| display_name | varchar(100) | N | 표시 이름 |
| timezone | varchar(64) | Y | 기본 타임존, 기본값 `Asia/Seoul` |
| created_at | timestamptz | Y | 생성 시각 |
| updated_at | timestamptz | Y | 수정 시각 |

인덱스:

- unique `email`

### 9.2 calendars

캘린더 기본 정보 테이블이다.

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| id | uuid | Y | 캘린더 ID |
| owner_user_id | uuid | N | 소유자 ID, 시스템 캘린더는 null 가능 |
| type | varchar(30) | Y | `USER`, `SYSTEM_HOLIDAY` |
| name | varchar(80) | Y | 캘린더 이름 |
| color | varchar(7) | Y | HEX 색상, 예: `#2563EB` |
| description | varchar(500) | N | 설명 |
| visibility | varchar(30) | Y | `PRIVATE`, `LINK`, `PUBLIC` |
| is_readonly | boolean | Y | 읽기 전용 여부 |
| is_default | boolean | Y | 사용자 기본 캘린더 여부 |
| system_key | varchar(80) | N | 시스템 캘린더 식별자, 예: `KR_HOLIDAYS` |
| created_at | timestamptz | Y | 생성 시각 |
| updated_at | timestamptz | Y | 수정 시각 |
| deleted_at | timestamptz | N | 소프트 삭제 시각 |

제약:

- `type = SYSTEM_HOLIDAY`이면 `is_readonly = true`
- `type = SYSTEM_HOLIDAY`이면 `owner_user_id`는 null 허용
- 사용자 캘린더는 `owner_user_id` 필수
- `color`는 HEX 6자리 형식
- `name`은 공백 제거 후 1자 이상

인덱스:

- `owner_user_id`
- `type`
- unique partial `system_key` where `system_key is not null`

### 9.3 calendar_members

사용자별 캘린더 권한과 구독 상태를 저장한다.

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| id | uuid | Y | 멤버십 ID |
| calendar_id | uuid | Y | 캘린더 ID |
| user_id | uuid | Y | 사용자 ID |
| role | varchar(30) | Y | `OWNER`, `VIEWER`, `EDITOR`, `ADMIN` |
| subscription_status | varchar(30) | Y | `SUBSCRIBED`, `HIDDEN`, `UNSUBSCRIBED` |
| display_order | integer | N | 목록 정렬 순서 |
| created_by | uuid | N | 공유를 추가한 사용자 |
| created_at | timestamptz | Y | 생성 시각 |
| updated_at | timestamptz | Y | 수정 시각 |

제약:

- `(calendar_id, user_id)` unique
- 일반 캘린더는 반드시 1명의 `OWNER`를 가진다.
- 시스템 공휴일 캘린더는 모든 사용자에게 `VIEWER`로 구독된다.
- `OWNER`는 권한 강등 또는 삭제할 수 없다.

인덱스:

- `user_id`
- `calendar_id`
- `(user_id, subscription_status)`

### 9.4 calendar_share_invitations

공유 초대를 이메일 기반으로 처리하려는 경우 사용하는 테이블이다. MVP에서 대상 사용자가 이미 가입되어 있어야 한다면 이 테이블 없이 즉시 `calendar_members`를 만들 수 있다.

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| id | uuid | Y | 초대 ID |
| calendar_id | uuid | Y | 캘린더 ID |
| inviter_user_id | uuid | Y | 초대한 사용자 |
| invitee_email | varchar(255) | Y | 초대 대상 이메일 |
| role | varchar(30) | Y | `VIEWER`, `EDITOR`, `ADMIN` |
| token_hash | varchar(255) | N | 초대 링크 토큰 해시 |
| status | varchar(30) | Y | `PENDING`, `ACCEPTED`, `EXPIRED`, `CANCELED` |
| expires_at | timestamptz | N | 만료 시각 |
| accepted_user_id | uuid | N | 수락 사용자 |
| accepted_at | timestamptz | N | 수락 시각 |
| created_at | timestamptz | Y | 생성 시각 |
| updated_at | timestamptz | Y | 수정 시각 |

MVP 권장:

- 1차 구현은 가입된 사용자 이메일로 즉시 공유한다.
- 초대 링크 방식은 이후 확장으로 둔다.

### 9.5 events

일정 테이블이다.

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| id | uuid | Y | 일정 ID |
| calendar_id | uuid | Y | 소속 캘린더 ID |
| created_by | uuid | N | 생성 사용자, 시스템 일정은 null 가능 |
| updated_by | uuid | N | 마지막 수정 사용자 |
| source | varchar(30) | Y | `USER`, `SYSTEM_HOLIDAY` |
| source_key | varchar(120) | N | 외부 원본 식별자 |
| title | varchar(120) | Y | 제목 |
| description | text | N | 설명 |
| location | varchar(255) | N | 장소 |
| starts_at | timestamptz | Y | 시작 시각 |
| ends_at | timestamptz | Y | 종료 시각 |
| timezone | varchar(64) | Y | 일정 기준 타임존 |
| is_all_day | boolean | Y | 종일 여부 |
| status | varchar(30) | Y | `CONFIRMED`, `CANCELED` |
| created_at | timestamptz | Y | 생성 시각 |
| updated_at | timestamptz | Y | 수정 시각 |
| deleted_at | timestamptz | N | 소프트 삭제 시각 |

제약:

- `ends_at > starts_at`
- 사용자 일정은 `created_by` 필수
- 공휴일 일정은 `source = SYSTEM_HOLIDAY`
- 공휴일 일정은 소속 캘린더가 `SYSTEM_HOLIDAY` 타입이어야 한다.
- 읽기 전용 캘린더에는 사용자 API로 일정을 생성할 수 없다.

인덱스:

- `(calendar_id, starts_at, ends_at)`
- `(starts_at, ends_at)`
- `(source, source_key)`
- partial index `deleted_at is null`

종일 일정 처리:

- 저장은 `timestamptz`로 통일한다.
- 종일 일정은 사용자 타임존 기준 시작일 00:00, 종료일 다음날 00:00으로 저장한다.
- 예: 2026년 5월 24일 하루 종일이면 `starts_at = 2026-05-24T00:00:00+09:00`, `ends_at = 2026-05-25T00:00:00+09:00`.

### 9.6 calendar_display_preferences

사용자별 캘린더 표시 설정을 멤버십과 분리하고 싶을 때 사용하는 선택 테이블이다. MVP에서는 `calendar_members.subscription_status`에 포함해도 된다.

## 11. 캘린더 공유 권한 정책

### 11.1 권한 단계

| 권한 | 일정 조회 | 일정 생성 | 일정 수정 | 일정 삭제 | 캘린더 정보 수정 | 공유 관리 | 캘린더 삭제 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| VIEWER | 가능 | 불가 | 불가 | 불가 | 불가 | 불가 | 불가 |
| EDITOR | 가능 | 가능 | 가능 | 가능 | 불가 | 불가 | 불가 |
| ADMIN | 가능 | 가능 | 가능 | 가능 | 가능 | 가능 | 불가 |
| OWNER | 가능 | 가능 | 가능 | 가능 | 가능 | 가능 | 가능 |

### 11.2 기본 정책

- 캘린더 생성자는 자동으로 `OWNER`가 된다.
- `OWNER`는 캘린더당 1명으로 시작한다.
- MVP에서는 소유권 이전을 제공하지 않는다.
- `ADMIN`은 공유 사용자 추가, 권한 변경, 제거를 할 수 있다.
- `ADMIN`은 `OWNER`를 제거하거나 권한 변경할 수 없다.
- `EDITOR`는 일정 CRUD만 가능하다.
- `VIEWER`는 일정과 캘린더 기본 정보를 조회할 수 있다.
- 시스템 공휴일 캘린더는 모든 사용자에게 `VIEWER`로 제공된다.

### 11.3 공유 대상 정책

- MVP 1차는 이미 가입된 사용자에게만 공유한다.
- 공유 대상은 이메일로 검색한다.
- 존재하지 않는 이메일이면 `USER_NOT_FOUND` 에러를 반환한다.
- 자기 자신에게 공유할 수 없다.
- 이미 공유된 사용자를 다시 추가하면 권한 변경으로 처리한다.
- 탈퇴했거나 비활성화된 사용자는 공유 대상에서 제외한다.

### 11.4 공개상태와 공유 권한의 관계

캘린더의 `visibility`는 외부 접근 가능성을 나타내고, `calendar_members.role`은 명시적 사용자 권한을 나타낸다.

MVP 정책:

- `PRIVATE`: 명시적으로 공유된 사용자만 접근 가능
- `LINK`: 링크를 가진 사용자가 보기 가능
- `PUBLIC`: 공개 조회 가능

단, MVP 1차 구현에서는 `PRIVATE`를 기본으로 실제 동작 범위로 삼는다. `LINK`, `PUBLIC`은 데이터와 설정 UI만 준비하거나, 구현 시 조회 권한만 허용한다. 공개 캘린더에도 일정 생성/수정은 명시적 멤버 권한이 있어야 한다.

### 11.5 읽기 전용 시스템 캘린더 정책

- `SYSTEM_HOLIDAY` 타입 캘린더는 사용자 수정이 불가하다.
- 공휴일 일정은 사용자 API로 생성, 수정, 삭제할 수 없다.
- 사용자는 공휴일 캘린더의 표시 여부만 변경할 수 있다.
- 운영 배치만 공휴일 캘린더 이벤트를 생성하거나 갱신할 수 있다.
- 공휴일 캘린더 이름, 색상은 운영 정책으로만 변경한다.

### 11.6 권한 확인 순서

API 요청마다 다음 순서로 확인한다.

1. 로그인 여부 확인
2. 대상 캘린더 또는 일정 존재 여부 확인
3. 삭제된 리소스 여부 확인
4. 시스템 읽기 전용 리소스 여부 확인
5. 사용자 멤버십 확인
6. 요청 작업에 필요한 권한 확인
7. 입력값 검증
8. 데이터 변경

보안을 위해 존재하지 않는 리소스와 접근 권한이 없는 리소스는 상황에 따라 모두 `404`로 응답할 수 있다. 단, 클라이언트 UX가 필요한 경우 명확한 `403`을 사용할 수 있다.

## 12. 대한민국 공휴일 배치 수집 정책

### 12.1 수집 대상

### 12.2 수집 주기

권장 스케줄:

- 매일 새벽 03:00 KST에 현재 연도와 다음 연도 공휴일 수집
- 매월 1일 새벽 03:30 KST에 전년도, 현재 연도, 다음 연도 재수집
- 운영자 수동 실행 시 특정 연도 재수집

이유:

- 공휴일 데이터는 자주 바뀌지 않지만, 대체공휴일 지정이나 임시공휴일 지정 가능성이 있다.
- 다음 연도 데이터가 미리 필요한 사용자 경험을 고려한다.
- 전년도 재수집은 데이터 보정과 운영 실수 복구용이다.

### 12.3 수집 소스

우선순위:

1. 공공데이터 기반 특일/공휴일 API
2. 운영자가 승인한 보조 데이터 소스
3. 수동 보정 데이터

MVP에서는 하나의 공식 소스를 기준으로 구현한다. 외부 API 장애가 있을 수 있으므로 원본 응답 요약과 실패 로그를 저장한다.

### 12.4 정규화 규칙

외부 원본을 내부 모델로 변환할 때 다음 규칙을 적용한다.

- 날짜는 `YYYY-MM-DD` 형식의 `date`로 저장한다.
- 이름은 사용자가 보는 공휴일명으로 저장한다.
- 현지명은 한국어 이름으로 저장한다.
- 공휴일 이벤트는 종일 일정으로 생성한다.
- 시작 시각은 해당 날짜 `00:00:00 Asia/Seoul`이다.
- 종료 시각은 다음 날짜 `00:00:00 Asia/Seoul`이다.
- 같은 날짜에 여러 공휴일명이 있으면 각각 이벤트로 만들 수 있지만, MVP에서는 동일 날짜 중복 표시를 피하기 위해 이름이 다를 때만 별도 생성한다.

### 12.5 중복 방지

중복 판단 키:

- `country_code`
- `holiday_date`
- `source_key`

이벤트 반영 키:

- 공휴일 캘린더 ID
- `source = SYSTEM_HOLIDAY`
- `source_key`

처리:

- 원본이 이미 있으면 이름, 원본 payload, 마지막 수집 시각을 갱신한다.
- 연결된 이벤트가 있으면 제목과 날짜를 갱신한다.
- 연결된 이벤트가 없으면 새 이벤트를 생성한다.
- 외부 소스에서 사라진 기존 공휴일은 즉시 삭제하지 않고 `skipped` 또는 별도 검토 대상으로 남기는 방식을 권장한다.

### 12.6 삭제와 변경 처리

공휴일은 정책 변경으로 수정될 수 있다. MVP에서는 안전하게 다음 정책을 사용한다.

- 같은 `source_key`의 날짜 또는 이름이 변경되면 기존 이벤트를 갱신한다.
- 특정 연도 재수집 시 원본에 없는 기존 공휴일은 자동 삭제하지 않는다.
- 자동 삭제 대신 운영 검토 플래그를 남기는 구조를 향후 추가할 수 있다.
- 임시공휴일처럼 갑자기 추가되는 데이터는 다음 배치에서 추가된다.

### 12.7 배치 상태

배치 상태는 다음 중 하나다.

- `RUNNING`: 실행 중
- `SUCCESS`: 전체 성공
- `PARTIAL_SUCCESS`: 일부 항목 실패
- `FAILED`: 전체 실패

상태 판단:

- 외부 조회 실패: `FAILED`
- 조회 성공 후 일부 항목 저장 실패: `PARTIAL_SUCCESS`
- 조회 성공 및 모든 항목 반영 성공: `SUCCESS`
- 실행 도중 예기치 않은 예외: `FAILED`

### 12.8 실패 로그 저장 정책

실패 로그에는 다음을 저장한다.

- 실행 ID
- 실패 단계
- 에러 코드
- 에러 메시지
- 외부 API 상태 코드
- 외부 응답 요약
- 실패한 원본 payload
- 재시도 가능 여부

민감정보 정책:

- API 키, 인증 토큰, Authorization 헤더는 저장하지 않는다.
- 외부 응답 전문이 너무 크면 요약만 저장한다.
- 원본 payload는 필요한 필드만 저장하거나 크기 제한을 둔다.

### 12.9 재시도 정책

권장:

- 네트워크 오류, 5xx 응답, 타임아웃은 재시도 가능
- 4xx 인증/권한 오류는 재시도 불가
- 데이터 검증 실패는 항목 단위 실패로 기록하고 다음 항목 처리

MVP 재시도:

- 배치 내부 자동 재시도는 최대 2회
- 재시도 간격은 3초, 10초
- 최종 실패 시 `holiday_sync_failures`에 기록

### 12.10 공휴일 캘린더 초기화

서비스 초기 세팅 시 다음 작업을 수행한다.

1. `calendars`에 `SYSTEM_HOLIDAY`, `system_key = KR_HOLIDAYS` 캘린더를 생성한다.
2. 기본 이름은 `대한민국 공휴일`로 설정한다.
3. 기본 색상은 일반 사용자 캘린더와 구분되는 색상으로 설정한다.
4. 기존 모든 사용자에게 `VIEWER`, `SUBSCRIBED` 멤버십을 생성한다.
5. 이후 신규 사용자는 가입 또는 첫 접속 시 자동 구독한다.

## 13. 입력 검증

### 13.1 캘린더 검증

이름:

- 필수
- 앞뒤 공백 제거
- 1자 이상 80자 이하
- 제어문자 불가

색상:

- 필수
- HEX 형식만 허용
- 예: `#2563EB`
- 투명도 포함 HEX는 MVP에서 제외

설명:

- 선택
- 500자 이하
- 스크립트 실행이 되지 않도록 출력 시 escape 처리

공개상태:

- `PRIVATE`, `LINK`, `PUBLIC` 중 하나
- 기본값 `PRIVATE`

### 13.2 공유 검증

이메일:

- 필수
- 이메일 형식
- 가입된 사용자여야 함
- 자기 자신 불가

권한:

- `VIEWER`, `EDITOR`, `ADMIN` 중 하나
- `OWNER`는 공유 API로 부여 불가

대상 캘린더:

- 존재해야 함
- 삭제되지 않아야 함
- 시스템 캘린더가 아니어야 함
- 요청자가 `OWNER` 또는 `ADMIN`이어야 함

### 13.3 일정 검증

제목:

- 필수
- 앞뒤 공백 제거
- 1자 이상 120자 이하

캘린더:

- 필수
- 존재해야 함
- 삭제되지 않아야 함
- 읽기 전용이 아니어야 함
- 사용자가 `EDITOR` 이상이어야 함

시작/종료:

- 필수
- ISO 8601 형식
- `endsAt`은 `startsAt`보다 이후여야 함
- 종일 일정은 날짜 단위로 정규화
- 너무 긴 일정 제한은 MVP에서 1년 이하 권장

타임존:

- 기본값 `Asia/Seoul`
- IANA timezone 형식
- MVP에서는 사용자 기본 타임존과 `Asia/Seoul` 중심으로 처리

장소:

- 선택
- 255자 이하

설명:

- 선택
- 서버 저장 제한은 별도 정책에 따르되, MVP는 5,000자 이하 권장

### 13.4 공휴일 배치 검증

연도:

- 1970 이상 2100 이하
- 운영자 수동 실행 시 필수

국가 코드:

- MVP는 `KR`만 허용

원본 데이터:

- 날짜 필수
- 이름 필수
- 날짜 파싱 실패 시 항목 실패로 기록
- 공휴일 여부가 false인 데이터는 저장하지 않거나 `skipped` 처리

## 14. 에러 처리

### 14.1 공통 에러 코드

| 코드 | HTTP 상태 | 설명 |
| --- | --- | --- |
| UNAUTHORIZED | 401 | 로그인이 필요함 |
| FORBIDDEN | 403 | 권한 없음 |
| NOT_FOUND | 404 | 리소스 없음 |
| VALIDATION_ERROR | 400 | 입력값 오류 |
| CONFLICT | 409 | 중복 또는 상태 충돌 |
| READONLY_CALENDAR | 403 | 읽기 전용 캘린더 변경 시도 |
| CALENDAR_MEMBER_NOT_FOUND | 404 | 공유 멤버 없음 |
| HOLIDAY_SYNC_FAILED | 500 | 공휴일 배치 실패 |
| INTERNAL_ERROR | 500 | 알 수 없는 서버 오류 |

### 14.2 캘린더 에러 처리

- 없는 캘린더 조회: `NOT_FOUND`
- 권한 없는 캘린더 조회: `NOT_FOUND` 또는 `FORBIDDEN`
- 시스템 캘린더 수정: `READONLY_CALENDAR`
- 캘린더 이름 누락: `VALIDATION_ERROR`
- 색상 형식 오류: `VALIDATION_ERROR`
- 삭제된 캘린더 수정: `NOT_FOUND`

### 14.3 공유 에러 처리

- 대상 사용자 없음: `NOT_FOUND`
- 자기 자신 공유: `VALIDATION_ERROR`
- 권한 없는 공유 변경: `FORBIDDEN`
- `OWNER` 제거 시도: `CONFLICT`
- 시스템 캘린더 공유 변경 시도: `READONLY_CALENDAR`
- 이미 공유된 사용자 추가: 신규 생성 대신 권한 변경으로 처리하거나 `CONFLICT`를 반환한다. MVP 권장은 권한 변경 처리다.

### 14.4 일정 에러 처리

- 종료 시간이 시작 시간보다 빠름: `VALIDATION_ERROR`
- 제목 없음: `VALIDATION_ERROR`
- 읽기 전용 캘린더에 일정 생성: `READONLY_CALENDAR`
- 보기 권한으로 일정 수정: `FORBIDDEN`
- 공휴일 일정 삭제: `READONLY_CALENDAR`
- 조회 기간 누락 또는 과도하게 큼: `VALIDATION_ERROR`

조회 기간 제한:

- 월간/주간/일간 조회는 클라이언트 뷰 기준 기간으로 요청한다.
- API는 최대 조회 기간을 400일 이하로 제한하는 것을 권장한다.

### 14.5 공휴일 배치 에러 처리

- 외부 API 타임아웃: 실패 로그 저장 후 재시도
- 외부 API 5xx: 실패 로그 저장 후 재시도
- 외부 API 인증 실패: 재시도하지 않고 실패
- 응답 파싱 실패: 실패
- 일부 항목 검증 실패: `PARTIAL_SUCCESS`
- DB 저장 실패: 실패 로그 저장 후 배치 실패 또는 부분 실패

운영자가 봐야 할 최소 정보:

- 언제 실행됐는지
- 어떤 연도를 수집했는지
- 몇 건을 받았는지
- 몇 건이 생성/갱신됐는지
- 어떤 단계에서 실패했는지
- 재시도 가능한지

## 16. 수용 기준

### 16.1 캘린더 관리

- 사용자는 새 캘린더를 생성할 수 있다.
- 사용자는 캘린더 이름, 색상, 설명, 공개상태를 수정할 수 있다.
- 사용자는 본인 소유 캘린더를 삭제할 수 있다.
- 사용자는 캘린더별 표시 여부를 변경할 수 있다.
- 공휴일 캘린더는 삭제하거나 수정할 수 없다.

### 16.2 일정 관리

- 사용자는 편집 권한 이상인 캘린더에 일정을 생성할 수 있다.
- 사용자는 접근 가능한 캘린더의 일정을 조회할 수 있다.
- 사용자는 편집 권한 이상인 캘린더의 일정을 수정할 수 있다.
- 사용자는 편집 권한 이상인 캘린더의 일정을 삭제할 수 있다.
- 보기 권한 사용자는 일정 수정/삭제 버튼을 사용할 수 없다.
- 공휴일 일정은 수정/삭제할 수 없다.

### 16.3 PC 화면

- PC 캘린더 페이지에서 월/주/일 뷰를 전환할 수 있다.
- PC 캘린더 페이지에서 오늘, 이전, 다음 이동이 가능하다.
- PC 좌측 목록에서 캘린더 표시 여부를 변경할 수 있다.
- PC에서 일정 생성, 상세 조회, 수정, 삭제가 가능하다.
- 일정은 소속 캘린더 색상으로 표시된다.

### 16.4 모바일 화면

- 모바일 캘린더 페이지에서 월/주/일 뷰를 전환할 수 있다.
- 모바일 월간 뷰에서 날짜 선택과 선택 날짜 일정 목록 조회가 가능하다.
- 모바일에서 캘린더 필터를 열고 표시 여부를 변경할 수 있다.
- 모바일에서 일정 생성, 상세 조회, 수정, 삭제가 가능하다.
- 화면 폭이 좁아도 주요 버튼과 텍스트가 겹치지 않는다.

### 16.5 공유 권한

- 소유자는 캘린더를 다른 사용자에게 공유할 수 있다.
- 소유자는 공유 사용자의 권한을 변경할 수 있다.
- 소유자는 공유 사용자를 제거할 수 있다.
- 보기 권한 사용자는 일정 조회만 가능하다.
- 편집 권한 사용자는 일정 CRUD가 가능하지만 캘린더 설정과 공유 설정은 변경할 수 없다.
- 관리자 권한 사용자는 캘린더 정보와 공유 설정을 변경할 수 있다.
- 관리자 권한 사용자는 소유자를 제거하거나 권한 변경할 수 없다.

### 16.6 공휴일 배치

- 대한민국 공휴일 시스템 캘린더가 존재한다.
- 모든 사용자는 공휴일 캘린더를 기본 구독한다.
- 공휴일 배치가 특정 연도의 공휴일을 수집한다.
- 수집된 공휴일은 종일 일정으로 표시된다.
- 배치 실행 이력이 저장된다.
- 실패 시 실패 로그가 저장된다.
- 같은 배치를 다시 실행해도 중복 공휴일 이벤트가 생성되지 않는다.

## 17. 주요 리스크

### 17.1 권한 로직 복잡도

리스크:

- 캘린더 권한과 일정 권한이 섞이면 보안 문제가 생길 수 있다.

대응:

- 일정 권한은 항상 소속 캘린더 멤버십 기준으로 판단한다.
- API마다 권한 확인 함수를 공통화한다.
- 권한별 테스트 케이스를 먼저 작성한다.

### 17.2 공휴일 데이터 변경

리스크:

- 임시공휴일, 대체공휴일 등으로 데이터가 변경될 수 있다.

대응:

- 현재 연도와 다음 연도를 매일 재수집한다.
- upsert 구조로 중복을 방지한다.
- 삭제는 자동 처리하지 않고 보수적으로 접근한다.

### 17.3 모바일 UX 과밀

리스크:

- 월간 달력에 일정 제목을 모두 표시하면 모바일에서 읽기 어렵다.

대응:

- 모바일 월간 뷰는 점 표시와 선택 날짜 리스트 중심으로 설계한다.
- 상세한 일정 정보는 목록과 상세 화면에서 제공한다.

### 17.4 시간대와 종일 일정

리스크:

- 종일 일정 저장 방식이 불명확하면 날짜가 하루 밀릴 수 있다.

대응:

- MVP 기본 타임존은 `Asia/Seoul`로 둔다.
- 종일 일정은 시작일 00:00, 종료일 다음날 00:00으로 저장한다.
- API와 UI에서 같은 정규화 규칙을 사용한다.

### 17.5 조회 성능

리스크:

- 일정이 많아지면 월간 조회 성능이 떨어질 수 있다.

대응:

- 기간 조건과 캘린더 ID 조건에 맞는 인덱스를 둔다.
- API 조회 기간을 제한한다.
- 삭제된 일정은 partial index 또는 조건으로 제외한다.

### 17.6 공개상태 정책

리스크:

- `PUBLIC`, `LINK` 공개상태를 완전 구현하면 접근 정책이 복잡해진다.

대응:

- MVP 1차에서는 `PRIVATE` 중심으로 구현한다.
- `LINK`, `PUBLIC`은 필드와 UI만 준비하거나 조회 전용으로 제한한다.
- 공개 캘린더의 수정은 명시적 멤버 권한이 있어야만 허용한다.

## 18. 오픈 결정 사항

1. MVP에서 `LINK`, `PUBLIC` 공개상태를 실제 접근 가능 기능으로 구현할지, 설정값만 저장할지 결정이 필요하다.
2. 공유 대상이 가입된 사용자만 가능한지, 미가입 이메일 초대까지 포함할지 결정이 필요하다.
3. 캘린더 소유권 이전을 MVP에서 제외할지 확정이 필요하다.
4. 일정의 캘린더 이동을 허용할지 결정이 필요하다.
5. 삭제는 소프트 삭제로 통일할지, 일부 데이터는 하드 삭제할지 결정이 필요하다.
6. 공휴일 배치의 공식 데이터 소스를 어떤 API로 확정할지 결정이 필요하다.
7. 공휴일 원본에서 사라진 항목을 자동 삭제할지, 운영 검토 대상으로 둘지 결정이 필요하다.
8. 운영자용 공휴일 배치 이력 화면을 MVP UI에 포함할지 결정이 필요하다.
9. 모바일 주간 뷰를 시간표로 만들지, 날짜 스트립과 목록으로 단순화할지 결정이 필요하다.
10. 일정 설명의 최대 길이와 HTML 허용 여부를 확정해야 한다.
11. 캘린더 색상 팔레트를 고정 제공할지, 사용자가 자유 HEX를 입력할지 결정해야 한다.
12. 한 사용자당 생성 가능한 캘린더 수 제한을 둘지 결정해야 한다.
13. 한 캘린더당 공유 가능한 사용자 수 제한을 둘지 결정해야 한다.

## 20. 기술 스택 및 Oracle Cloud 배포 전략

### 20.1 최종 추천 스택

MVP의 기본 기술 스택은 다음 조합을 권장한다.

| 영역 | 추천 기술 | 역할 |
| --- | --- | --- |
| Web | `Next.js App Router` | PC/모바일 웹 캘린더 UI, 관리자 화면, 로그인 화면 |
| Backend API | `Spring Boot` | 인증/인가, 캘린더/일정 API, 공유 권한, 공휴일 배치 |
| Database | `PostgreSQL` | 사용자, 캘린더, 일정, 공유 권한, 배치 이력 저장 |
| Mobile App | `React Native + Expo` | 추후 iOS/Android 앱 구현 |
| Deploy | `OCI Compute VM + Docker Compose + nginx reverse proxy` | 1차 MVP 운영 배포 |

이 조합은 웹 MVP 개발 속도, 백엔드 운영 안정성, 모바일 앱 확장성을 함께 고려한 선택이다.

### 20.2 추천 이유

#### Next.js App Router를 웹 프레임워크로 사용

현재 작업공간에는 `Next.js` 기반 프로젝트가 여러 개 있어 기존 구현 패턴과 개발 경험을 재사용하기 쉽다. 캘린더 서비스의 PC/모바일 웹 화면, 관리자 화면, 로그인 화면을 빠르게 구성하기에 적합하다.

다만 핵심 비즈니스 로직은 `Server Actions`에 강하게 묶지 않는다. 나중에 모바일 앱이 같은 기능을 사용해야 하므로, 웹 UI는 API를 호출하는 클라이언트로 두고 캘린더/일정/공유 권한 로직은 백엔드 API에 둔다.

#### Spring Boot를 Backend API로 사용

캘린더 서비스는 단순 화면보다 백엔드 규칙이 중요하다. 특히 다음 기능은 서버에서 일관되게 검증해야 한다.

- 캘린더별 공유 권한
- 일정 생성/조회/수정/삭제 권한
- 읽기 전용 시스템 공휴일 캘린더 보호
- 대한민국 공휴일 배치 실행
- 배치 실행 이력과 실패 로그 저장
- 트랜잭션과 예외 처리

`Spring Boot`는 인증/인가, 입력 검증, 트랜잭션, 스케줄링, 운영 로그 구성이 성숙해 업무형 백엔드에 적합하다. 공휴일 배치는 MVP에서는 `@Scheduled`로 시작하고, 배치 규모가 커지면 `Spring Batch`로 확장할 수 있다.

#### PostgreSQL을 기본 데이터베이스로 사용

캘린더, 일정, 공유 권한, 배치 로그는 관계형 모델에 잘 맞는다. `PostgreSQL`은 기간 조회, 인덱스, JSON 원본 payload 저장, 트랜잭션 처리에 적합하다.

운영 환경에서는 `OCI Database with PostgreSQL` 같은 관리형 서비스를 우선 검토한다. 비용을 우선해야 하는 초기 MVP에서는 OCI VM 내부의 Docker 기반 PostgreSQL로 시작할 수 있지만, 이 경우 백업과 복구 정책을 반드시 별도로 준비한다.

#### React Native + Expo를 추후 모바일 앱 프레임워크로 사용

나중에 앱을 만들 계획이 있으므로 모바일은 `React Native + Expo`를 권장한다. 웹이 `Next.js`와 TypeScript 기반이면 API 타입, DTO, 일부 검증 규칙, 도메인 용어를 모바일과 맞추기 쉽다.

`Flutter`도 모바일 완성도는 높지만, 웹과 언어/타입/모델 공유가 약해진다. 별도 모바일 전문 팀이 있거나 Flutter 경험이 강한 경우가 아니라면 MVP 이후 확장 경로로는 `React Native + Expo`가 더 자연스럽다.

### 20.3 제외 또는 비추천 판단

#### Next.js 단일 앱

`Next.js` 단일 앱은 MVP를 가장 빠르게 만들 수 있다. 하지만 API Route, Server Action, UI 로직이 섞이면 모바일 앱을 붙일 때 API 계약을 다시 정리해야 할 가능성이 크다.

이 서비스는 나중에 앱 확장이 예정되어 있으므로, 처음부터 웹 전용 서버 구조가 아니라 독립 Backend API 구조를 잡는 편이 안전하다.

#### NestJS

`NestJS`는 TypeScript 단일 스택을 유지할 수 있고 REST API 구조화에도 강점이 있다. 그러나 현재 요구사항에서는 공유 권한, 트랜잭션, 배치, 실패 로그, 운영 안정성이 중요하다.

현재 작업공간에 이미 `Spring Boot` 백엔드 프로젝트가 있고, 공휴일 배치와 권한 검증을 안정적으로 운영해야 하므로 기본 추천은 `Spring Boot`로 둔다.

#### Flutter

`Flutter`는 iOS/Android 앱 품질을 높게 만들 수 있는 좋은 선택이다. 하지만 웹과 같은 TypeScript 생태계를 공유하기 어렵고, API 타입과 도메인 모델을 별도로 관리해야 한다.

캘린더 서비스의 1차 목표가 웹 MVP이고 이후 앱 확장이라면 `React Native + Expo`가 더 낮은 전환 비용을 가진다.

### 20.4 Oracle Cloud 배포 전략

#### 1차 MVP 배포

MVP는 단순하고 예측 가능한 구조로 시작한다.

- OCI Compute VM 1대 생성
- VM 내부에 Docker와 Docker Compose 설치
- `web` 컨테이너로 `Next.js` 실행
- `api` 컨테이너로 `Spring Boot` 실행
- `nginx` 컨테이너 또는 VM nginx로 reverse proxy 구성
- `/` 요청은 `web`으로 전달
- `/api/v1/*` 요청은 `api`로 전달
- HTTPS 인증서 적용

이 구조는 초기 비용과 운영 복잡도를 낮추면서도 웹과 API를 명확히 분리할 수 있다.

#### 운영 전 필수 보강

공개 운영 전에는 다음 항목을 추가한다.

- TLS 인증서 자동 갱신
- `web` 헬스체크 엔드포인트
- `api` 헬스체크 엔드포인트
- DB 백업 정책
- 배치 실패 로그 조회 경로
- 서버 로그 보관 정책
- 배포 롤백 절차
- 환경변수와 비밀값 분리

#### 확장 단계

트래픽이나 운영 요구가 커지면 다음 순서로 확장한다.

1. OCI Load Balancer를 추가해 외부 진입점을 분리한다.
2. `web`과 `api` 컨테이너를 각각 독립 VM 또는 Container Instance로 분리한다.
3. 배치가 중복 실행되지 않도록 DB lock 또는 별도 scheduler를 도입한다.
4. 컨테이너 수와 배포 대상이 많아지면 `OKE` 이전을 검토한다.

`OCI Container Instances`는 서버 관리 부담을 줄이고 컨테이너 단위 실행에 적합하다. `OKE`는 여러 서비스, 자동 확장, 배포 전략, 네트워크 정책이 필요해질 때 검토한다. MVP 단계에서 바로 Kubernetes를 도입하지는 않는다.

### 20.5 모바일 앱 대비 API 원칙

웹과 모바일 앱은 같은 Backend API를 사용한다. API는 처음부터 웹 전용이 아니라 제품 공용 API로 설계한다.

기본 원칙:

- API prefix는 `/api/v1`을 사용한다.
- 인증, 캘린더, 일정, 공유, 공휴일 배치 API를 웹 UI와 분리한다.
- 웹은 API의 첫 번째 클라이언트이며, 모바일 앱은 두 번째 클라이언트로 붙인다.
- API 응답 포맷은 웹과 앱에서 동일하게 사용한다.
- 권한 검증은 클라이언트가 아니라 서버에서 최종 처리한다.
- 날짜/시간은 ISO 8601과 IANA timezone을 기준으로 주고받는다.
- 모바일 앱의 오프라인/동기화 확장을 위해 주요 리소스에는 `id`, `createdAt`, `updatedAt`을 포함한다.

권장 API 그룹:

- `/api/v1/auth`
- `/api/v1/me`
- `/api/v1/calendars`
- `/api/v1/events`
- `/api/v1/calendar-members`
- `/api/v1/holiday-sync-runs`

이 구조를 따르면 PC 웹, 모바일 웹, iOS/Android 앱이 같은 도메인 규칙과 API 계약을 공유할 수 있다.
